### ⛅ 만들게 된 계기

우선 저는 naver, notion, velog 라는 플랫폼을 이용해 공부 기록과 프로젝트 관련 기록을 해왔습니다.
이미 잘 구현되어있는 이러한 플랫폼을 두고 어쩌다가 블로그를 만들게 되었을까요?

#### 1. 개발자로서의 작업물
블로그 자체가 나만의 작업물이 될 수도 있고, 프론트엔드 개발자로서 나만의 블로그가 있다는 건 꽤나 멋지다고 생각했습니다.🤩

#### 2. 내 스타일대로 Customizing
FE 개발자로서 스타일링에 관심을 가져야 한다는 말을 많이 들었고, 나만의 블로그를 통해 개성을 표현할 좋은 기회라고 생각했습니다.
기존 블로그 플랫폼에서도 커스터마이징 기능을 제공하지만, 기본 설정에 제한이 있어 자유로운 표현이 어렵다는 점에서 직접 블로그를 제작하기로 했습니다.

> 처음 프로젝트에 참여했을 때는 팀원들과 의미만 잘 전달되면 된다고 생각해, 협업에 필요한 기술 용어나 개념을 깊이 있게 이해하려 하지 않았습니다. 그러나 점차 제대로 이해하고 소통이 잘 되는 개발자가 되기 위해 용어 정리의 필요성을 느꼈고, 나만의 기록을 남길 수 있는 블로그를 만들고 싶다는 생각이 들었습니다.
이번에 Next.js로 기술 블로그를 제작하며, 블로그를 제작하는 과정에 대한 기록과 프로젝트에서 자주 쓰이는 용어들을 체계적으로 이해하고 꾸준히 기록하는 것을 목표로 하고 있습니다.

-----

### 1. 기능 정리
블로그에서 구현하고자 하는 기능들을 정리해보았습니다.

**필수 기능**
- 게시글 목록 페이지(전체, 카테고리별)
- 게시글 상세 페이지
- 댓글 기능

**부가 기능**
- 다크/라이트 모드
- 최상단, 최하단 이동 버튼
- scroll progress status bar
- 검색 엔진 최적화(SEO)

### 2. 기술 스택 정하기
이 블로그에 적용된 기술 스택은 다음과 같습니다.

#### ⚙️ Framework: Next.js
프론트엔드 생태계에서 가장 활발히 사용되는 프레임워크 중 하나가 Next.js라고 생각합니다. 이미 여러 프로젝트에서 사용해 익숙하기도 하고, 점점 Next.js를 도입하는 회사들도 늘어나는 추세입니다.
최근 업데이트 된 app router 방식으로 블로그를 구축하여, pages router 보다 간결하게 라우팅을 설정하고 관리할 수 있다는 이점이 있었습니다.

#### 🎨 Style: Tailwind CSS
여러 스타일 라이브러리를 사용해봤지만, Tailwind CSS는 비교적 직관적이며 유연하게 커스터마이징할 수 있어 선택했습니다. 커스텀 테마나 컴포넌트 스타일링을 쉽게 설정할 수 있어 생산성 높은 코드를 작성할 수 있었고, 직관적인 클래스 네이밍 덕분에 유지보수 측면에서도 유리하다고 느꼈습니다.

#### 📃 Contents 작성: MDX
글은 markdown 문법으로 작성된 MDX 파일을 프로젝트 내부에서 렌더링하는 방식으로 구현했습니다.

#### 📚 Contents 파싱: @next/mdx + next-mdx-remote
- @next/mdx: MDX 파일을 사용하기 위한 설정 패키지
- next-mdx-remote: MDX 콘텐츠의 파싱이나 직렬화 작업

작년까지 많은 개발자들이 사용했던 **Contentlayer** 를 사용해보고 싶었는데, 최근 커뮤니티 지원이 중단되어 유지보수 및 업데이트가 멈춘 상태라고해서 다른 라이브러리를 선택하게 되었습니다.

#### 📢 Publish: Vercel
Next.js는 Vercel에서 만든 프레임워크로 호환성이 매우 높은 cloud 플랫폼입니다.

### 🔍 용어 정리

#### ✏️ 마크다운(Markdown)
**Markdown** 은 간단한 문법을 통해 텍스트 기반 문서를 쉽게 작성할 수 있게 하는 마크업 언어입니다. 복잡한 HTML 태그를 몰라도 제목, 목록, 코드 블록 등을 손쉽게 표현할 수 있다는 장점이 있습니다. 다만 Markdown 파일은 브라우저에서 바로 해석되지 않기 때문에 웹이 표시하려면 HTML로 변환하는 별도의 _컴파일 또는 렌더링 과정_ 이 필요합니다!

#### ✏️ 정적 페이지(Static Page)
**정적 페이지** 는 서버에서 미리 생성된 HTML 파일을 클라이언트에 그대로 전달하는 방식입니다. 이러한 페이지는 사용자에게 동일한 내용을 제공하며, 미리 생성된 HTML 파일을 전달하기 때문에 서버 요청에 대한 응답 속도가 빠릅니다. 또한, 완성된 HTML 페이지를 제공하므로 검색 엔진이 쉽게 크롤링할 수 있어 SEO에도 유리합니다.

#### ✏️ 동적 페이지(Dynamic Page)
**동적 페이지** 는 사용자의 요청에 따라 서버에서 실시간으로 데이터를 가져와 HTML을 생성하는 방식입니다. 사용자가 요청할 때마다 서버가 데이터를 조회하고 페이지를 생성하므로, 상황에 따라 내용이 달라질 수 있다는 특징이 있습니다. 보통 데이터베이스와 연결되어 사용자 맞춤형 정보나 실시간 데이터를 제공하며, 정적 페이지보다 로딩 시간이 더 길 수 있다는 단점이 있습니다.
