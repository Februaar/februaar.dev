> 프로젝트 초기에는 팀원 간의 대략적인 의미 전달에 집중하여, 기술 용어들을 깊이 있기 이해하지 않고 겉핥기식으로 소통했다. 하지만 나중에 기본적인 용어조차 제대로 설명하지 못하는 자신을 발견하면서, 용어 정리의 필요성을 절실히 느꼈다.

그런 경험을 통해 기술 블로그를 만들기로 했고, 이를 시작하기 전 가장 중요한 용어들부터 정리해보았다.

## 📌 용어 정리

### ✅ 마크다운(Markdown)

Markdown은 간단한 문법을 통해 텍스트 기반 문서를 작성할 수 있도록 고안된 마크업 언어이다. 복잡한 HTML 태그 없이도 문서를 구성할 수 있어, 작성의 편의성과 가독성을 높여준다. 파일 확장자는 .md 또는 .markdown을 사용하며, 흔히 볼 수 있는 README 파일, 블로그 포스트, 기술 문서 등에 널리 사용된다.

### ✅ 정적 페이지(Static Page)

정적 페이지는 서버에서 미리 생성된 HTML 파일을 클라이언트에 그대로 전달하는 방식이다.

특징
➡️ 한 번 만들어진 페이지는 사용자가 누구든 동일한 내용을 제공하며, 이미 만들어진 HTML 파일을 전달하기 때문에 서버 요청에 대한 응답 시간이 매우 빠르다는 점이 있고, 검색 엔진이 쉽게 크롤링할 수 있는 완성된 HTML 페이지를 제공하므로 SEO에 유리하다.

### ✅ 동적 페이지(Dynamic Page)

동적 페이지는 서버에서 사용자의 요청에 따라 실시간으로 콘텐츠가 생성되거나 변경되는 페이지이다. 사용자가 페이지에 요청할 때마다 서버에서 새로운 데이터를 가져와 HTML을 만들어 전달하는 방식이다.

특징
➡️ 사용자가 요청하는 시점에 서버에서 데이터를 가져와 페이지를 생성하고, 따라서 페이지의 내용은 상황에 따라 달라질 수 있다. 쉽게 생각하면 보통 데이터베이스와 연결되어 사용자별로 다른 데이터를 제공하거나, 실시간으로 변화하는 데이터를 기반으로 페이지가 생성되기 때문에 콘텐츠가 변경 가능하다는 특징이 있는 것이다. 그리고 정적 페이지와 가장 큰 대비를 보이는 곳은 바로 응답 시간이 느리다는 것인데, 서버에서 데이터를 가져와 HTML을 생성하는 과정이 필요하기 때문에 정적 페이지보다 로딩 시간이 더 걸릴 수 있다.

Next.js로 블로그를 만들기 위해 여러 라이브러리를 비교하는 과정에서 많은 시간이 소요되었다. 이 과정에서 고려했던 주요 라이브러리들과 그 중 선택한 이유를 정리해보았다.

## 📌 라이브러리 선택

최종적으로는 @next/mdx 패키지를 사용하기로 결정했다.

### 🔍 contentlayer

작년까지는 많은 개발자들이 Contentlayer를 사용하여 MDX 파일을 처리했지만, 최근 들어 커뮤니티 지원이 중단되어 유지보수 및 업데이트가 멈춘 상태이다. 이로 인해 장기적으로 안정적인 선택이 아니라는 판단이 들었다.

### 🔍 next-mdx-remote

next-mdx-remote는 MDX 파일을 유연하게 커스터마이징할 수 있는 장점이 있다. 특히 외부 소스의 MDX 콘텐츠를 로드하여 동적으로 사용할 수 있는 기능이 유용하다. 하지만 로컬에서 MDX 파일을 직접 가져와 처리하는 경우, 설정이 번거로울 수 있다는 점에서 블로그와 같은 프로젝트에서는 다소 비효율적이라고 생각이 들었다.

### 🔍 @next/mdx

Next.js와 원활하게 통합되고 안정적인 지원을 받을 수 있다는 점이 가장 큰 장점이었다. 별도의 커스텀 설정 없이도 MDX 파일을 쉽게 관리할 수 있어 블로그처럼 정적인 콘텐츠를 다루는 프로젝트에 최적화되었다고 생각했다.

@next/mdx는 Next.js와의 호환성 덕분에 개발과 유지보수가 용이하여, 블로그 프로젝트에 적합하다고 판단해 최종적으로 선택하게 되었다.
